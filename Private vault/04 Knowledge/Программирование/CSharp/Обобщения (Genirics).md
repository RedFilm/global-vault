Обобщения - это конструкции, которые позволяют писать код, который будет одинаково работать с разными типами данных (общий код для разных типов данных), при этом сохраняя строгую типизацию. Одним из главных преимуществ обобщений является возможность избежать операций boxing и unboxing.

имеется метод, который меняет местами элементы a и b. В данном случае этот метод работает с типом int.

Но что, если возникнет необходимость работать с ещё каким-нибудь типом данных? Один из вариантов - перегрузка метода:

![[Private vault/04 Knowledge/Documents/6c61f6a50d238f0d6f30c25515eadb07_MD5.png]]

Но этот подход не всегда оптимальный, хорошо было бы иметь 1 универсальный метод, который умел бы работать с разными типами данных. Именно эту задачу и позволяет решить обобщение. Ниже представлен пример, как с помощью обобщений можно сделать данный метод универсальным:

![[Private vault/04 Knowledge/Documents/131a0a8c92fe161a3c5a3892bd1e547c_MD5.png]]

Стоит объяснить, как это работает. Хотя мы и создаём один метод для разных типов данных, «под капотом» происходит создание точно такого же метода, в который подставляется тип данных, с которым необходимо сейчас работать. Таким же образом для каждого используемого типа данных генерируется новый метод.

Среда разработки во время компиляции переводит c# код в IL-код (промежуточный код, в который переводятся языки платформы .NET), далее этот код, с помощью JIT (Just In Time) компилируется в байт-код. Метод Swap будет генерироваться в байт код для каждого типа данных, с которым будет необходимо работать. Если однажды байт-код был сгенерирован для определённого типа данных, то далее он будет использоваться повторно, а не генерироваться ещё раз.

List (обобщенные коллекции) является обёрткой над обычным массивом. В нём, благодаря использованию обобщений, мы также можем работать с различными типами данных.

Рассмотрим 2 примера и их сравнение в производительности. В первом случае использованы обобщения, во втором реализация с помощью типа object:

![[Private vault/04 Knowledge/Documents/c836b32d95c02bd69b29fc87883bdbbc_MD5.png]]

Результат теста показан ниже:

![[Private vault/04 Knowledge/Documents/f0a09983932864f2a25a487042c6e18c_MD5.png]]

Из теста видно, что скорость метода, реализованного с помощью обобщений гораздо выше. Связано это с операцией упаковки. Во втором методе идёт упаковка данных (из int в object), соответственно это занимает определённое время и задействует дополнительные ресурсы, а именно выделяется доп. память в управляемой куче для хранения упакованных данных. Для очистки кучи вызывается сборщик мусора, во время работы которого приложение полностью останавливается. Эти операции и объясняют большую разницу в производительности.

Ниже приведён ещё один пример. В нём реализованы 2 метода, первый из которых создаёт массив, использующий тип object, а второй - обобщения.

![[Private vault/04 Knowledge/Documents/fd70a64a24d0222e44b57c444db3fab6_MD5.png]]

Далее показан результат выполнения этих методов:

![[Private vault/04 Knowledge/Documents/d5b2a680f437db0f4ce12cfac1b3764a_MD5.png]]

В данном случае оба массива использовали память в управляемой куче, но при этом тот, который использовал реализацию с обобщениями всё равно выполнил те же действия быстрее. Почему? Объясняется это нюансами хранения массивов в управляемой куче.

Сама переменная массива представляет из себя ссылку на ячейку в управляемой куче, с которой начинаются данные. Стоит учесть, что данные массива хранятся непрерывно друг за другом. Если это значимые типы данных, то каждая ячейка хранит в себе непосредственно значения:

![[Private vault/04 Knowledge/Documents/aa385fe32914c4c823deed5c549dabef_MD5.png]]

Но если массив хранит в себе ссылочные типы, то каждая ячейка будет представлять из себя очередную ссылку на реальные данные:

![[Private vault/04 Knowledge/Documents/6246db5b418dca998eea19d2bcaf8dd5_MD5.png]]

В этом случае данные могут находиться где угодно. Проблема возникает во время вызова сборщика мусора, в этом случае ему недостаточно удалить последовательность ссылок,![[Private vault/04 Knowledge/Documents/7f81bb0458a972d99c6687c1cf4d5800_MD5.png]]

необходимо также удалить и данные, на которые эти ссылки указывают. Таким образом сборщику мусора потребуется гораздо больше времени на удаление этих объектов.
