# Пилот: план выполнения и шаблон отчета

## 1) Параметры пилота
- Количество клиентов: 1-2.
- Объем: 1 подразделение/цех на клиента.
- Длительность: 4-6 недель.
- Участники: авторы кейсов, эксперт-валидатор, руководитель пилота.

## 2) Цели пилота
- Проверить, что система сокращает время повторной диагностики.
- Подтвердить регулярность добавления кейсов в формате voice-first.
- Подтвердить готовность клиента к коммерческому продолжению.

## 3) План пилота по неделям

### Неделя 0 (подготовка)
- Назначить владельцев пилота с обеих сторон.
- Согласовать KPI, baseline и отчетность.
- Провести быстрый onboarding (60 минут).

### Неделя 1
- Запуск на ограниченном контуре.
- Сбор первых 20-30 кейсов.
- Ежедневный сбор обратной связи по UX и качеству поиска.

### Неделя 2-3
- Увеличение объема кейсов.
- Еженедельная калибровка тегов/структурирования.
- Устранение критичных проблем в процессе внесения знаний.

### Неделя 4-6
- Стабилизация использования.
- Сравнение KPI относительно baseline.
- Подготовка итогового pilot report и решения Go/No-Go.

## 4) Базовые метрики и формулы

### Метрики активности
- Количество созданных кейсов за неделю.
- Доля голосового ввода = voiceCases / totalCases.
- Доля активных пользователей = activeUsers / totalPilotUsers.

### Метрики качества поиска
- Search usefulness rate = helpfulSearches / totalSearches.
- Top-3 relevance (экспертная выборка).

### Метрики операционного эффекта
- Снижение времени повторной диагностики:
  - `(baselineMean - pilotMean) / baselineMean * 100%`
- Доля инцидентов, где применен существующий кейс.

### Метрики коммерческого эффекта
- Готовность к расширению (Да/Нет + условия).
- Оценка потенциального бюджета/модели оплаты.

## 5) Риски пилота и действия
- Низкая активность ввода -> ввести микро-регламент "1 кейс после каждого нетипового инцидента".
- Слабая релевантность -> еженедельная ревизия структуры и тегов.
- Сопротивление процессу -> champion внутри смены + короткие обучающие сессии.
- ИТ-ограничения -> заранее согласовать контур и доступы.

## 7) Шаблон Pilot Report (итог)

```text
1. Executive summary
- Цель пилота
- Достигнут/не достигнут целевой эффект

2. Контур пилота
- Площадка/подразделение
- Количество пользователей
- Длительность

3. KPI до/после
- Время повторной диагностики: X -> Y
- Полезность поиска: X%
- Объем базы кейсов: X
- Доля голосового ввода: X%

4. Ключевые кейсы успеха
- Кейc #1
- Кейc #2

5. Ограничения и доработки
- Что помешало достичь максимального эффекта
- Must-have улучшения перед масштабированием

6. Коммерческое решение
- Go/No-Go
- Условия перехода в контракт
- Предложенный объем расширения
```

## 8) Критерии Go/No-Go
- Go:
  - Снижение времени повторной диагностики >= 20%.
  - Полезность поиска >= 35%.
  - Клиент подтверждает расширение на соседние участки.
- No-Go:
  - Эффект ниже порога и нет признаков быстрого улучшения.
  - Низкая продуктовая дисциплина без перспективы исправления.
