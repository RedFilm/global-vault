# MVP PRD и технический blueprint

## 1) Цель MVP
Сократить время повторной диагностики производственных проблем за счет voice-first захвата кейсов и поиска похожих решений.

## 2) Область MVP (in/out)

### In scope
- Голосовой ввод кейса (web/mobile).
- Транскрибация речи в текст.
- Автоструктурирование кейса по полям.
- Ручное редактирование перед публикацией.
- Поиск похожих кейсов (semantic + фильтры).
- Ролевой доступ и журнал изменений.

### Out of scope
- Продвинутая аналитика с дашбордами уровня BI.
- Массовые интеграции (более 2 коннекторов).
- Автоматическое управление работами в CMMS.

## 3) Пользовательские роли
- Автор кейса (оператор/инженер): создает и редактирует собственные кейсы.
- Эксперт/валидатор: подтверждает качество кейсов.
- Руководитель: просматривает KPI и отчеты.
- Администратор: управляет ролями, справочниками и доступом.

## 4) Ключевые user stories
- Как инженер, я хочу надиктовать кейс за 60-90 секунд, чтобы быстро зафиксировать решение.
- Как начальник смены, я хочу найти похожие подтвержденные кейсы по симптомам, чтобы ускорить диагностику.
- Как эксперт, я хочу переводить кейс в статус "Проверено", чтобы команда использовала надежные знания.
- Как руководитель, я хочу видеть метрики использования и эффекта, чтобы оценивать ROI.

## 5) Критические сценарии (E2E)

### Сценарий A: Создание кейса
1. Пользователь нажимает "Записать кейс".
2. Система принимает аудио и делает ASR.
3. LLM выделяет структуру: проблема, контекст, действия, результат, риски.
4. Пользователь подтверждает/правит и публикует.
5. Система индексирует кейс для similarity-поиска.

### Сценарий B: Поиск решения
1. Пользователь голосом/текстом задает проблему.
2. Система извлекает признаки запроса (симптом, оборудование, условия).
3. Выполняется hybrid retrieval (BM25 + vector similarity).
4. Выдаются топ-5 кейсов с confidence и источником.
5. Пользователь открывает кейс и применяет решение.

## 6) Модель данных (минимум)

### Entity: Case
- caseId
- title
- problemDescription
- context (line, equipment, shift, environment)
- actionsTaken (list)
- result
- risksAndWarnings
- tags (equipmentType, failureType, rootCause)
- authorId
- status (Draft, Reviewed, Outdated)
- createdAt, updatedAt

### Entity: CaseRevision
- revisionId, caseId
- changedBy, changedAt
- changeSummary
- oldValue/newValue (audit)

### Entity: QueryLog
- queryId, userId, queryText
- retrievedCases[]
- selectedCaseId
- wasHelpful (boolean)

## 7) API-контракты (черновой минимум)
- `POST /cases/voice-upload` -> загрузка аудио, получение transcriptionId.
- `POST /cases/structure` -> структурирование текста в schema CaseDraft.
- `POST /cases` -> создание кейса.
- `PATCH /cases/{id}` -> правка кейса.
- `POST /search` -> поиск похожих кейсов.
- `POST /cases/{id}/review` -> перевод в Reviewed/Outdated.
- `POST /feedback/helpful` -> фидбек по полезности результата.

## 8) Архитектура MVP (логическая)
- Клиент: web (desktop + mobile-responsive).
- Backend API: auth, cases, search, audit.
- AI pipeline:
  - ASR (speech-to-text)
  - LLM-структурирование по JSON schema
  - embeddings для similarity.
- Storage:
  - relational DB для кейсов/ролей/аудита,
  - vector index для семантического поиска.

## 9) Нефункциональные требования
- Время создания кейса: <= 90 секунд (p50).
- Время ответа поиска: <= 3 секунд (p95).
- Доступность MVP в пилоте: >= 99%.
- Полный аудит изменения кейсов.
- Ролевой доступ на уровне организации/подразделения.

## 10) План сборки MVP (4 спринта)

### Sprint 1: Основа
- Auth + роли.
- CRUD кейсов (текстовый ввод).
- Черновой поиск по тегам/тексту.

### Sprint 2: Voice-first
- Запись и загрузка аудио.
- Транскрибация.
- Черновое автоструктурирование.

### Sprint 3: Similarity и качество
- Hybrid retrieval.
- Confidence и объяснение релевантности.
- Фидбек "полезно/не полезно".

### Sprint 4: Пилотная готовность
- Аудит и журнал изменений.
- Экспорт pilot metrics.
- Исправление UX-узких мест.

## 11) Критерии готовности MVP к пилоту
- Все критические E2E-сценарии работают стабильно.
- Не менее 80% тестовых кейсов корректно структурируются (с экспертной проверкой).
- Поиск возвращает релевантный кейс в топ-3 минимум в 60% тестовых запросов.
- Есть экспорт отчетов по базовым KPI пилота.
